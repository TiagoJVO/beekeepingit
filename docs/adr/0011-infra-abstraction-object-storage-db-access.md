## Context

[NFR-ARC-2](../../requirements/non-functional-requirements.md) requires the app not be
tightly coupled to a specific database technology or cloud/hosting environment.
`requirements/tech-stack.md` already commits to the concrete choices — **MinIO now,
S3-compatible → swap to cloud later**, and **Postgres via pgx + sqlc** (a typed query
layer, not raw provider calls) — and
[`docs/architecture/service-decomposition.md`](../architecture/service-decomposition.md) §7
names the same seam: _"object storage via an S3-compatible interface... and DB access via a
typed query layer... so the database/cloud can be swapped without touching domain logic."_
Nothing implemented that seam yet. This is also the **first Go code landing in the repo** —
`taskfiles/go.yml`'s own header comment anticipates `services/*/go.mod` as the discovery
convention.

**Scope boundary vs. #20** ("Shared Go service template"): #20 owns the full service
scaffold (health/readyz, structured logging, OTel, JWT/JWKS middleware, RFC 9457 error
format) and will import this package for its own "data-access layer demonstrated by a
sample endpoint" AC. This task stops at the **library + adapters + a documented
config-switch example** — it does not build an env/config _loading framework_ (that's #20's
AC); these packages take an explicit `Config` struct, keeping the two issues' scope
non-overlapping.

**"Broker" in #85's AC:** no message broker exists anywhere in `requirements/tech-stack.md`
or the built `infra/` — the AC's wording is forward-looking. Noted rather than silently
dropped: the same "config/secrets, never hardcoded" pattern applies whenever one is
introduced.

## Decision

Add a `services/shared` Go module (not a deployable service — a library other `services/*`
modules import), containing:

1. **`objectstore`** — a `Store` type wrapping [`minio-go/v7`](https://github.com/minio/minio-go),
   built from a `Config{Endpoint, AccessKey, SecretKey, UseSSL, Region}`. minio-go is itself a
   generic S3-compatible client, so the _same_ adapter code talks to MinIO today and AWS
   S3/another provider later purely by changing `Config` — no separate interface/mock layer
   needed on top.
2. **`dbaccess`** — `Connect(ctx, Config) (*pgxpool.Pool, error)` (fails fast: pings before
   returning), `Migrate(ctx, dsn, fs.FS)` running versioned migrations via
   [`goose`](https://github.com/pressly/goose)'s `Provider` API against a `database/sql`
   connection (`pgx/v5/stdlib` driver) — the app's own queries still go through the
   `pgxpool` pool from `Connect`. `dbaccess/sqlc/` ships one minimal reference migration +
   typed query set (`platform_example.items`) generated by `sqlc`, proving the full
   pgx+sqlc+goose pipeline end-to-end without inventing a fake domain model; it's a pattern
   for future services to copy, not a real feature.
3. A `README.md` showing two side-by-side `Config` literals (local dev vs. a differently
   hosted endpoint) driving the identical `New`/`Connect` calls — the documented proof that
   swapping is config-only. The integration tests (testcontainers-go, MinIO + Postgres
   modules) are the executable proof the adapters work end-to-end, against ephemeral
   containers rather than the dev cluster.

Located at `services/shared/` (not e.g. `platform/` or `pkg/`) because `taskfiles/go.yml`
already anticipated `services/*/go.mod` as the discovery convention, and
`requirements/tech-stack.md`'s backend section already names **"shared libs"** as the
cross-cutting concept distinct from domain services.

### golangci-lint v1 → v2 (unplanned but forced by this task)

Landing real dependencies (`pgx`, `minio-go`, `testcontainers-go`) pulled in transitive
`golang.org/x/{crypto,net,sync,sys,text}` and `grpc-gateway/v2` requiring Go ≥ 1.25.0.
golangci-lint v1's final release (1.64.8) was built with go1.24.1 and refuses to lint any
module targeting Go 1.25+ — not a choice specific to this package selection, just where the
Go ecosystem sits in mid-2026. [ADR-0008](0008-monorepo-tooling.md) already logged "v1 → v2
migration" as a follow-up; this task executes it now (via `golangci-lint migrate`, a
mechanical config-schema port — same linters, same behavior) rather than pinning the whole
dependency tree to increasingly old versions to keep an EOL linter binary working.
`mise.toml`'s `go` pin moves `1.24` → `1.25` for the same reason.

### CI workflow renamed `lint` → `ci`

Once `task ci` (which the CI workflow runs) started running `task test` too, a GitHub check
literally named `lint` gating merges on test results was misleading. Renamed the workflow file
`.github/workflows/lint.yml` → `ci.yml` (`name: lint` → `ci`, job id `lint:` → `ci:`) and updated
`main`'s branch-protection required status check from `lint` to `ci` in the same change, so no
PR is left blocked on a check name that no longer exists.

## Consequences

**Positive**

- NFR-ARC-2's abstraction is now real, not just declared intent — both the storage and DB
  seams exist and are tested end-to-end.
- `services/shared` gives #20 (and every future domain service) a ready-made, tested
  data-access/object-storage layer to import instead of re-deriving pgx/sqlc/goose wiring
  per service.
- Unblocks Go 1.25+ dependencies repo-wide, not just for this module — the golangci-lint v2
  migration was going to be needed by the next Go module regardless.

**Negative / risks**

- `dbaccess/sqlc/schema.sql` duplicates the "up" DDL from
  `dbaccess/migrations/00001_create_example_items.sql` (sqlc applies migration files
  sequentially, so it would otherwise also "see" the down migration's `DROP TABLE`). Both
  files must be kept in sync — acceptable for one tiny demo table; a real per-service schema
  would need the same convention documented once, not per-service.
- `services/shared` isn't yet wired into a `go.work` — the first domain service that needs
  to import it will need to add one (or a `replace` directive), since no second module exists
  yet to prove that path.
- golangci-lint v2's config schema differs from v1's (migrated automatically via
  `golangci-lint migrate`); any contributor referencing v1 docs/examples should use v2 docs
  going forward.

## Alternatives considered

- **`platform/` or `pkg/` as the top-level location** instead of `services/shared/`.
  Rejected: `taskfiles/go.yml` and `tech-stack.md` already anticipated `services/*` +
  "shared libs" as the vocabulary; introducing a second top-level Go home would fragment
  discovery for no benefit.
- **A hand-rolled interface + separate mock implementation for `objectstore`** (e.g. an
  `ObjectStore` interface with MinIO and in-memory implementations). Rejected: minio-go is
  already a generic S3-compatible client — an extra interface layer would be indirection
  without a second real implementation to justify it (NFR-ARC-2 is satisfied by the client's
  own provider-agnosticism, not by a hand-rolled abstraction on top of it).
- **Pin the dependency tree to pre-1.25 versions** to avoid touching golangci-lint. Rejected
  per the discussion above — fragile (pins security-relevant `x/crypto`/`x/net`) and only
  delays a migration ADR-0008 already flagged as needed.
- **Build #20's full config-loading framework here too**, so #85 ships something more
  "complete." Rejected to keep the two issues' scope non-overlapping — see Context.
